# 个人实训总结报告

> AI 辅助编程与工程化实训（居家自学版）
> 项目名称：宠物日常护理与健康追踪助手
> 课题编号：课题一

## 一、项目概述

本次实训我独立完成了"宠物日常护理与健康追踪助手"全栈小程序项目，前端采用 uni-app + Vue 3，后端采用 Spring Boot + Spring Data JPA + MySQL。项目实现了首页仪表盘（体重/体温变化曲线、今日待办提醒）、护理日志页（喂食/洗澡/疫苗/驱虫记录、时间线展示、标签筛选）、健康报告页（月度报告、健康评分）三个独立视图，以及新增宠物页面。后端实现了新增健康日志、月度统计、今日待办三个核心 API，并利用 Spring @Scheduled 定时任务自动扫描过期提醒。

## 二、遇到的最大困难及解决过程

开发过程中我遇到了多个技术难题，其中最大的困难是**前后端数据联调中的空指针异常（NPE）与图表渲染问题**。

第一个典型问题是后端 `StatisticsService` 中的 NPE。在测试健康报告页时，月度统计接口持续报错 `Cannot invoke "Pet.getId()" because "Reminder.getPet()" is null`。通过分析 error.txt 中的后端控制台日志，我发现根因是数据库中存在 `pet_id` 为 null 的脏数据（早期新增待办时未正确关联宠物），而代码中 `reminderRepository.findAll().stream().filter(r -> r.getPet().getId()...)` 在 `getPet()` 返回 null 后直接调用 `getId()` 导致 NPE。解决过程分两步：先清理数据库脏数据，再将查询逻辑改为 `findByPetIdAndCareType(petId, CareType.VACCINE)` 直接按宠物 ID 查询，从根源上避免了 null 问题。

第二个典型问题是 ECharts 图表无法显示。最初使用 uni-app 的 renderjs 方式引入 ECharts，但在 H5 环境下极不稳定，图表经常空白。经过反复排查，发现两个关键问题：一是使用了 `<canvas>` 标签作为图表容器，而 ECharts 实际需要普通的 `<div>` DOM 元素；二是每次进入页面才动态加载 ECharts CDN 导致加载过慢。最终解决方案是在 `App.vue` 的 `onLaunch` 中预加载 ECharts，容器改为 `<view>`（渲染为 div），并增加 `waitForECharts()` 轮询等待机制确保 ECharts 就绪后再渲染图表。

此外，前端 API 参数传递方式也曾导致接口 404——前端用 `data: { petId }` 发送参数，但后端用 `@RequestParam` 接收查询参数，两者不匹配。修复方法是将前端 API 改为 URL 查询参数形式 `?petId=`。

## 三、对 AI 辅助编程的心得体会

本次实训让我深刻体会到了 AI 辅助编程的强大与局限。

**AI 的优势**在于：能够快速生成项目脚手架和完整的功能代码，大幅提升开发效率；在遇到报错时，只要提供完整的错误日志，AI 能精准定位问题根因并给出修复方案；对于 ECharts 配置、Spring Boot 注解等需要大量样板代码的场景，AI 能一步到位生成高质量配置。

**AI 的局限**在于：生成的代码并非完美无缺，经常存在潜在 bug（如 NPE、参数传递不匹配），需要开发者具备调试和验证能力；AI 对项目整体架构的把控不如人类，例如 `PetInfoController` 直接调用 Repository 跳过 Service 层的问题，AI 不会主动指出；AI 生成的代码需要开发者理解后才能放心使用，盲目信任会导致难以维护的代码堆砌。

我的经验是：**AI 是强大的编程助手，但不能替代开发者的思考**。每次使用 AI 生成的代码后，我都会仔细阅读、理解逻辑、测试验证，遇到问题主动排查 error.txt 等日志，再将问题精准地反馈给 AI 修复，形成"人工分析→AI 生成→人工验证→AI 修复"的高效闭环。

## 四、个人在项目中的具体贡献

本项目由我个人独立完成，具体贡献包括：

1. **需求分析与技术选型**：根据课题要求确定 uni-app + Spring Boot 技术栈，设计 3 张数据库表（pet / health_record / reminder）和枚举类型（CareType / MentalState）。

2. **后端开发**：实现 3 个核心 API（新增健康日志含数据校验、月度统计含平均值与波动率计算、今日待办），实现 @Scheduled 定时任务（每天 6:00 扫描过期提醒），配置 MySQL 数据库与 CORS 跨域。

3. **前端开发**：实现 4 个页面（首页仪表盘、护理日志、健康报告、新增宠物），使用 ECharts 绘制体重/体温变化曲线（含均值标线、渐变填充），自研轻量级 petStore 全局状态管理实现多页面宠物切换联动，实现待办事项完整 CRUD。

4. **问题排查与修复**：独立分析 error.txt 日志，修复后端 NPE、前端图表不显示、API 参数不匹配、宠物切换不联动等多个复杂 bug。

5. **项目优化**：优化健康报告页布局（健康评分算法、2×2 统计卡片网格），优化 ECharts 加载性能（预加载 + 轮询等待），数值统一保留两位小数。

## 五、深度思考与展望

通过本次实训，我认识到**工程规范比功能实现更重要**。一个能跑通的项目只是起点，代码的可维护性、分层架构的清晰度、异常处理的完备性才是衡量工程质量的核心标准。当前项目仍有改进空间：应引入 Bean Validation 规范数据校验、增加 @ControllerAdvice 全局异常处理、将 JPA 内存计算改为 SQL 聚合查询提升性能、增加单元测试保障代码质量。

未来如果继续迭代，我希望增加宠物图片上传功能（替代当前的 emoji 头像）、增加多宠物对比分析、引入消息推送实现真正的"主动提醒"，以及配置 CI/CD 实现自动化部署。
